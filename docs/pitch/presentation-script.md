### Block B — How Leo Works: Agents, Tools, and the AI (3 min)

> Now — what is actually happening inside the app when you talk to Leo?
>
> First, an important distinction.
>
> Leo is **not a chatbot**. It is an **AI agent**.
>
> What is the difference?
> A chatbot gives you text. You ask, it answers. That is all it can do.
> An agent can **think, decide, and act**. It has tools. It picks which tool to use. It can chain multiple tools together.
>
> Simple way to think about it: a chatbot is like a search engine — you type, you get a result. An agent is like an employee — it understands what you need, goes and gets the data, does the work, and comes back with the result.
>
> Everything you saw — the smart transfers, the spending charts, the stock analysis, the proactive coaching — that was the agent deciding what to do. We did not program "if user says X, do Y." The AI figured it out on its own.

**[SHOW architecture diagram]**

```
User sends a message
    → Goes to our backend
    → Backend sends it to the AI model
    → AI decides: what tools do I need?
    → Calls tools: get_transactions, show_spending_chart...
    → Gets the data back
    → Generates a response + interactive widget
    → Frontend shows it to the user
```

> The way this works is called **function calling**. We defined 10 tools for Leo — what each one does, what information it needs, what it gives back. The AI model decides when to use which tool.
>
> For example — when Sofia asked *"How are my investments doing?"* — the AI called `get_portfolio_data` to fetch her holdings, then checked market news, then called `show_stock_widget` to display the result. Three tools, chained automatically.
>
> Leo's 10 tools:
> - `get_account_balance` — account data
> - `get_recent_transactions` — transaction history
> - `get_portfolio_data` — investment holdings
> - `show_stock_widget` — stock chart with analysis
> - `show_transfer_widget` — money transfer
> - `show_spending_chart` — spending breakdown
> - `show_achievement` — badge animations
> - `show_savings_goal` — savings progress
> - `start_quiz` — launch a quiz
> - `navigate_to_screen` — go to a screen
>
> One more important thing: Junior and Adult mode use the **same AI**, the **same tools**. The only difference is a set of instructions called a **system prompt** — it changes Leo's personality. Junior gets simple language and gamification. Adult gets professional analysis. No code changes needed — just different instructions.

### Block C — Enterprise AI: How a Large Company Can Use This (3 min)

> Now I want to zoom out — away from the app — and talk about what this means for a large organization.
>
> The architecture you just saw — an AI agent with tools, connected to data sources — that is a pattern. And it works far beyond a banking assistant.

**[PAUSE]**

> **Implementation in an existing system.**
>
> You do not rebuild your app. You add a layer on top.

```
Existing App
    └── Existing APIs (any internal system)
            └── NEW: AI Layer (model + function calling)
                    └── NEW: Interface (chat, dashboard, or automation)
```

> Three steps:
> 1. Add an interface — could be a chat overlay, a dashboard, or even a background automation
> 2. Connect the AI layer to existing APIs — read access only
> 3. All actions go through existing confirmation and security flows
>
> The AI model is swappable. We used OpenAI for prototyping. In production, you use a self-hosted model inside your own infrastructure. Change the endpoint — everything else stays. **No data leaves your servers.**

**[PAUSE — new topic]**

> **MCP — Model Context Protocol.**
>
> This is an open standard for connecting AI agents to external systems. Think of it as a **universal plug**.
>
> You have internal systems — monitoring, ticket tracking, customer data, compliance tools. With MCP, you connect any of them to an AI agent. Same standard interface, any system. No custom integration code for each one.
>
> That means you can build one agent architecture — and plug in as many data sources as you need. Monitoring data, HR systems, financial data, operational dashboards — all through the same plug.

**[PAUSE — new topic]**

> **Skills and Hooks — enterprise safety.**
>
> In a regulated environment, you need control over what the AI can and cannot do.
>
> **Skills** are reusable procedures. They tell the agent *how* to handle specific tasks — step by step. "When giving financial advice, always add a disclaimer." "When accessing customer data, follow this process." Skills are stored as files in your repository. You standardize behavior across the whole organization.
>
> **Hooks** are policy enforcement. Before the agent uses a tool, a hook can approve or deny it. After it acts, a hook logs what happened. Full audit trail. Full compliance control.
>
> In banking: hooks ensure the agent never accesses data it should not see. Skills ensure it follows regulatory rules. And every action is logged and traceable.
>
> These are not ideas for the future — they are part of the open standard today.

# Q&A PREPARATION

## Expected Questions & Answers

### Business Questions

**Q: How much would it cost to implement this at ING?**
> Modular architecture. Start with just the chat layer — lowest cost entry point. Junior app is standalone. Full integration depends on existing API infrastructure, but the tech is standard.

**Q: What about ongoing AI costs?**
> ~1-3 cents per conversation at current scale. Enterprise pricing or self-hosted model at ING scale. Function-calling architecture works with any large language model — not locked to OpenAI.

**Q: How do you handle AI hallucinations or wrong advice?**
> Three safeguards: (1) Leo works with the user's own data — no invented numbers. (2) All actions require confirmation — Leo suggests, never executes alone. (3) Anomaly detection is based on pattern comparison against the user's own history.

### Technical Questions

**Q: Can this work with ING's existing backend?**
> Yes. API layer. Read access to accounts and transactions. No changes to core banking systems.

**Q: Why OpenAI and not a self-hosted model?**
> Prototyping speed. For production: any model works — Llama, Mistral, ING-hosted. Function calling interface is model-agnostic.

**Q: What is MCP? Why does it matter?**
> Model Context Protocol — an open standard for connecting AI agents to external systems. Like a universal plug. Instead of custom integrations for every data source, MCP gives the agent standardized access. If ING has an internal system, you can connect it to an AI agent through MCP without rebuilding anything.

**Q: What is the difference between an agent and a chatbot?**
> A chatbot gives text answers — you ask, it responds. An agent can think, decide, and act. It has tools, picks which one to use, and chains them together. Leo does not just answer questions — it fetches data, runs analysis, shows interactive widgets, and initiates actions. All autonomously.

**Q: What are skills and hooks?**
> Skills are reusable procedures — they tell the agent *how* to do specific tasks in a standardized way. Hooks are policy enforcers — they can approve or deny agent actions and log everything for audit. Together they give enterprise control: the agent works within defined boundaries, and every action is traceable.

**Q: How does the proactive coaching work technically?**
> The system prompt instructs Leo to analyze transaction patterns when the user opens the app or starts a chat. It compares current spending against historical averages and flags anomalies. For time-based alerts (rent due dates), it uses transaction recurrence data.

**Q: Data security with external AI?**
> Prototype uses OpenAI API. Production: self-hosted model within ING infrastructure. Swap the endpoint. No customer data leaves the building.

**Q: How does voice input work?**
> Browser speech-to-text API transcribes voice to text. The text goes to Leo as a normal message. No additional AI cost for voice — it is a frontend feature.

**Q: Can this agent approach work beyond Leo?**
> Yes. The pattern — agents, function calling, MCP, skills, hooks — is general-purpose. Connect any internal system: ticket tracker, monitoring, compliance tools. Same architecture. Same approach.

### Feature Questions

**Q: What about the real debit card for kids?**
> The junior account has a real ING card for real purchases. The virtual money system is separate — for learning and experimenting. Parents control the real card through spending limits and category restrictions.

**Q: Can parents see everything?**
> Parents see learning progress, quiz scores, achievements, and savings goals. They do not see private conversations with Leo. They control limits and rules. Trust goes both ways.

**Q: How does anomaly detection work?**
> Leo compares each transaction against the user's spending history. It looks for: duplicate subscriptions (same service, different accounts), transactions significantly above category averages, sudden category spikes, and recurring charges with unexpected amount changes.

**Q: Languages?**
> Leo responds in whatever language you write in. Built into the AI — zero extra development. Tested with Ukrainian for international users in Germany.

**Q: What about the leaderboard — can kids cheat?**
> XP is earned through verifiable actions — quiz answers, investment decisions, savings deposits. The AI generates unique quiz questions each time. School rankings require school verification.
