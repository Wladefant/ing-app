# PART 4: LEGAL & TRUST (2 min)

## Mateo (1 min)

> When it comes to AI in banking, the biggest question is: **can I trust it?**
>
> We designed Leo with three rules:
>
> **Rule 1: Your data stays yours.**
> Leo uses only your own transaction data — anonymized and encrypted.
> No data is shared with third parties. No open internet learning.
>
> **Rule 2: Leo suggests — you decide.**
> Leo can recommend actions, but it **never** executes without your approval.
> You are always in control.
>
> **Rule 3: Full compliance.**
> Leo is built to follow **GDPR** and the **EU AI Act**.
> Every recommendation is explainable. You can always ask *"Why?"* and Leo will tell you.

## Berkay (1 min)

> For minors, we add extra protection:
>
> - **Parental consent** is required to activate
> - **Age verification** at every step
> - **Simplified language** — no complex financial jargon
>
> And parents can see what their children are learning — without accessing their private data.
>
> The idea is simple: **trust through transparency.**

# PART 5: TECHNICAL DEEP DIVE (10 min)

## Vladimir

> Now let me show you **how** we built this.

### What is Vibe Coding? (2 min)

> We used an approach called **vibe coding**.
>
> Traditional software development works like this:
> 1. Designers spend weeks in Figma making static screens
> 2. Developers spend weeks turning those screens into code
> 3. Product managers review, find problems, and the cycle repeats
>
> With vibe coding, we skip step one.
>
> Instead of static designs, we **immediately build working prototypes** — using AI-assisted code generation.
>
> The result:
> - What you see is not a design — it is a **real, working application**
> - We can test and iterate in hours, not weeks
> - Every screen you saw today is functional — not a mockup

### Tech Stack (2 min)

> Here is what powers ING Leo:
>
> **Frontend:**
> - **React 19** with TypeScript — the same technology used by Meta, Netflix, and Airbnb
> - **Vite** — fast build tool, instant hot reload during development
> - **TailwindCSS** — utility-first styling for consistent, responsive design
> - **Framer Motion** — smooth animations (like the birthday transition you saw)
> - **Recharts** — the financial charts and spending visualizations
>
> **Backend:**
> - **Node.js with Express** — lightweight, fast API server
> - **PostgreSQL** database via **Drizzle ORM** — type-safe data access
> - **OpenAI API** with function calling — this is the brain of Leo
>
> **Key point for ING:** This stack is modern, well-supported, and easy to integrate with existing bank infrastructure.

### How Leo's AI Works (3 min)

> Let me explain how Leo actually thinks.
>
> When a user sends a message, here is what happens:

**[SHOW or describe the flow]**

> 1. The message goes to our backend
> 2. We send it to OpenAI with **function calling** enabled
> 3. The AI decides: does it need to use a tool, or just answer?
> 4. If it needs data — it calls a function: `get_account_balance`, `get_recent_transactions`, `get_portfolio_data`
> 5. With that data, it generates a response — plus any widgets
> 6. The frontend renders the response with interactive elements
>
> The important concept is **function calling**.
>
> Leo has access to **10 tools**:
> - `show_stock_widget` — display stock information
> - `show_transfer_widget` — initiate a money transfer
> - `start_quiz` — launch a learning quiz
> - `show_spending_chart` — visualize expenses
> - `get_portfolio_data` — fetch investment data
> - `navigate_to_screen` — move the user to a specific app section
> - and more
>
> The AI **decides on its own** which tools to use — based on what the user asks.
>
> This means Leo is not a simple chatbot with scripted answers.
> It is an **agent** that can take actions inside the banking app.

### How This Could Work at ING (2 min)

> For implementation at ING, the architecture is modular:
>
> **The AI layer is separate from the banking layer.**
>
> Leo does not touch core banking systems directly.
> It communicates through an API layer — the same way any frontend feature would.
>
> This means:
> - Leo can be added **on top** of existing infrastructure
> - Core banking systems stay untouched
> - The AI layer can be updated independently
> - You can start with one feature (like the chat) and expand over time
>
> **For the junior app:**
> - No real money is involved — only simulated data
> - This can be deployed as a standalone product
> - It connects to real market data APIs for stock prices
>
> **For the adult app:**
> - Connects to existing account and transaction APIs
> - The AI needs read access to transaction data — but never write access to accounts
> - All actions (transfers, investments) require explicit user confirmation

# Q&A PREPARATION

## Expected Questions & Answers

### Business Questions

**Q: How much would it cost to implement this at ING?**
> The architecture is modular. You could start with just the AI chat layer — which is the lowest-cost entry point. The junior app can be a standalone product. Full integration depends on existing API infrastructure, but the technology choices are deliberately standard and well-supported.

**Q: What about ongoing AI costs (OpenAI API)?**
> Current API costs are roughly 1-3 cents per conversation. At scale, ING could negotiate enterprise pricing or use a self-hosted model. The function-calling architecture works with any large language model — it is not locked to OpenAI.

**Q: How do you handle AI hallucinations or wrong financial advice?**
> Three safeguards: (1) Leo only works with the user's own data — it does not make up numbers. (2) All actions require user confirmation — Leo suggests, never executes alone. (3) Financial recommendations include disclaimers and are based on data analysis, not prediction.

### Technical Questions

**Q: Can this work with ING's existing backend systems?**
> Yes. The AI layer communicates through APIs. If ING already has REST APIs for accounts, transactions, and portfolio data — Leo can connect to them. No changes to core banking systems needed.

**Q: Why OpenAI and not a self-hosted model?**
> For prototyping speed, OpenAI was the fastest path. For production, the architecture supports any model — open source (Llama, Mistral) or ING's own hosted model. The function-calling interface is model-agnostic.

**Q: How do you ensure data security with an external AI provider?**
> In our prototype, data goes to OpenAI's API. For production at ING, the recommendation is to use a self-hosted model within ING's infrastructure. The architecture supports this — you just swap the API endpoint.

**Q: What happens when the AI is wrong?**
> Leo always shows its reasoning. Users can ask "Why?" for any recommendation. For financial decisions, we never auto-execute — the user always confirms. And for the junior app, everything is virtual — mistakes are literally the point of learning.

### Feature Questions

**Q: Can parents control what their children do in the app?**
> Yes. Parents must give consent to activate. They can see learning progress and quiz scores. They cannot see private chat conversations — to maintain trust with the child.

**Q: What about users who speak other languages?**
> Leo can respond in any language the user writes in — that is built into the AI. We also tested a Ukrainian language demo to show multilingual support for international users in Germany.

**Q: Can Leo work with voice input?**
> Yes, the chat supports voice input. The user speaks, the message is transcribed, and Leo responds as normal.
